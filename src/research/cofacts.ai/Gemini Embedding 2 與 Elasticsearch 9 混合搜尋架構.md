# **針對 Cofacts 多模態查核資料庫之 Gemini Embedding 2 與 Elasticsearch 9 混合搜尋架構評估報告**

## **核心系統架構之典範轉移與資訊戰防禦挑戰**

在當前高度複雜的數位資訊環境中，錯誤資訊與惡意謠言的傳播已不再侷限於單一文本形式，而是呈現出高頻率、跨模態（Cross-modal）的變異特徵。一個原始的文字謠言，可能在數小時內被截圖轉傳，隨後被惡意行為者製作成抖音（TikTok）短影音由真人朗讀，甚至透過生成式人工智慧（Generative AI）技術變造為無聲的深偽（Deepfake）影片。對於累積超過十年、資料量突破十萬筆的 Cofacts 查核資料庫而言，傳統基於光學字元辨識（OCR）、自動語音辨識（ASR）以及詞頻-逆文件頻率（TF-IDF）的檢索架構，已經面臨到系統性的效能瓶頸。

傳統架構在面對當前挑戰時，暴露出三個致命的脆弱節點。首先，TF-IDF 等詞彙檢索技術高度依賴字面精確比對，當造謠者刻意抽換同義詞或重組句型時，傳統系統便無法辨識其為已知謠言。其次，依賴逐字稿（Transcript）轉換的影音處理流程，會在其轉換過程中遺失大量的聲學特徵與語氣上下文；若遇到完全無文字的 AI 變造影像，且該影像經過二次轉檔或壓縮導致傳統雜湊（Hashing）演算法失效時，系統將完全失去比對能力。最後，跨模態檢索的斷層使得純文字版、截圖版與影片版的同一份謠言，在資料庫中被迫作為孤立的實體存在，無法在同一個搜尋結果列表中被統合呈現。

為了解決上述困境，將 Google 最新發布的 Gemini Embedding 2 模型與 Elasticsearch 9 的混合搜尋（Hybrid Search）技術進行深度整合，提供了一個革命性的技術路徑。Gemini Embedding 2 作為原生的多模態嵌入模型，能夠將文字、圖像、音訊與影片直接映射至同一個高維度語意向量空間中，徹底消除了模態之間的隔閡 。而 Elasticsearch 9 所具備的先進檢索器（Retrievers）架構，則能夠將向量語意相似度與傳統 TF-IDF/BM25 詞彙精準度進行完美的數學融合 。本報告將針對 Cofacts 資料庫的特殊需求，從跨模態語意對齊、長影音向量化策略、混合排名演算法最佳化，以及大規模歷史資料轉換的經濟成本等面向，進行極度詳盡的技術與可行性剖析。

## **Gemini Embedding 2 核心技術解析與跨模態語意對齊**

要評估 Gemini Embedding 2 運用於 Cofacts 系統的可行性，必須先深入理解其底層架構如何打破傳統資訊檢索的模態壁壘。過去的多模態嵌入技術（如 CLIP）通常採用雙編碼器架構，將視覺編碼器與文本編碼器分開訓練，並在流程的最後階段才透過對比學習進行對齊。這種作法忽略了不同模態在神經網路深層交互作用時所能產生的細微語意連結 。

### **原生多模態與交錯輸入處理**

Gemini Embedding 2 建立在 Gemini 基礎模型之上，從底層架構便繼承了真正的原生多模態理解能力 。這意味著它不再需要外部模型的輔助（例如先將音訊轉為文字，或將圖片生成描述），而是透過共享的 Transformer 架構，直接攝取原始的像素矩陣、音訊波形與文本標記，並將它們投影到一個單一的聯合嵌入空間（Joint Embedding Space）中 。

對於 Cofacts 面臨的「謠言變異」問題，這項技術帶來了決定性的突破。當系統需要處理抖音短影音中有人朗讀文字謠言的情況時，Gemini Embedding 2 能夠直接「吞噬」音訊波形，捕捉說話者的語意意圖、甚至是諷刺的語氣，而無需依賴容易產生錯誤的 ASR 逐字稿 。因為文字版謠言的語意與口述該謠言的音訊語意在向量空間中是高度重合的，這兩者將會被轉換成幾何距離極為接近的高維度向量，從而實現跨模態的同義判定 。同樣地，對於無文字的 AI 變造影像，模型提取的是畫面中的高階視覺語意特徵（如物體、場景、動作關係），而非底層的像素排列。因此，即使同一支影片經過惡意使用者的二次轉檔、壓縮或加入雜訊，其核心的視覺語意向量並不會產生劇烈偏移，系統依然能夠精準判定這與資料庫中現存的影片為同一則變造內容 。

此外，該模型支援交錯輸入（Interleaved Input）的能力，這完美契合了 Cofacts 使用者經由通訊軟體回報訊息的行為模式。使用者經常會同時傳送一張截圖加上一段文字提問，Gemini Embedding 2 允許在單次 API 請求中同時包含圖像與文字，模型能捕捉兩者之間的關聯性，產生一個代表聯合意義的單一嵌入向量 。

### **俄羅斯套娃表示學習（MRL）與維度最佳化**

在輸出規格上，Gemini Embedding 2 預設生成高達 3,072 維度的浮點數向量 。該模型實作了俄羅斯套娃表示學習（Matryoshka Representation Learning, MRL）技術，這是一種將最重要的語意資訊高度集中在向量陣列前端維度的架構設計 。這使得開發者可以根據儲存成本與運算效能的考量，動態地將向量維度截斷至 1,536、768 甚至更低，而不會引發災難性的語意流失 。

然而，針對 Cofacts 資料庫高度要求「近似重複（Near-duplicate）」偵測的情境，強烈建議維持 3,072 的最高維度。雖然降低維度可以節省 Elasticsearch 的記憶體消耗，但在處理高度相似的謠言變體，或是需要精細區分兩支僅有微小 AI 變造差異的影片時，向量後段所隱含的微觀語意特徵將發揮關鍵的區別作用 。因為 Cofacts 目前的總資料量約為十萬筆，屬於中等規模，維持全維度在現代硬體上的成本極低，完全沒有必要為了微小的儲存妥協而犧牲事實查核所需的最高精準度。

## **資料庫特性適配與向量化限制克服策略**

在將十萬筆圖文影音全面導入 Gemini Embedding 2 的過程中，必須嚴格應對模型 API 在各項模態上的技術限制，並針對 Cofacts 特殊的資料結構設計預處理管線（Preprocessing Pipeline）。

### **處理超大文本與網頁爬蟲資料**

Cofacts 資料庫中存在許多帶有超連結的訊息，系統會將網頁內容爬取回來作為參考依據。Gemini Embedding 2 將文本的輸入上下文限制大幅提升至 8,192 個標記（Tokens），相當於前一代模型的四倍之多 。8,192 個 Token 約等同於六千至八千個英文字或相當數量的中文字元，這對於絕大多數的假新聞文章、部落格內容或社群媒體長文來說已經綽綽有餘 。這意味著爬蟲抓取回來的網頁 DOM 文本，在經過基礎的 HTML 清洗後，幾乎可以直接作為單一文本輸入進行向量化，而無需設計複雜的文本分塊（Text Chunking）邏輯，這大幅降低了系統架構的複雜度並完整保留了文章的全域上下文 。

### **突破長影音長度限制的聚合策略**

Cofacts 面臨的最大技術挑戰，在於資料庫中的影片長度變異極大，部分影片長度達到 30 秒，而有些則長達 5 分鐘。根據 Gemini Embedding 2 API 的技術規格，單次請求處理影片的極限為：無音軌影片最長 120 秒，包含音軌的影片則最長限制為 80 秒 。因此，長達 5 分鐘（300 秒）的影片無法直接被單一 API 呼叫所攝取。

為了解決這項硬性限制，必須在後端實作「時序分塊與嵌入聚合（Temporal Chunking and Embedding Aggregation）」策略 。系統應將長影片程式化地切割為多個片段（例如每段 75 秒，並帶有 5 秒的重疊區域以防止語意截斷）。在取得這些分塊的向量後，有兩種主要的架構選擇來處理這些資料：

第一種是加權平均嵌入（Weighted Average / Mean Pooling），即將所有片段的向量進行數學平均，融合為一個代表整支影片的單一質心向量。雖然這能最小化 Elasticsearch 的儲存空間，但對於事實查核而言是極具風險的。這種做法會導致「語意崩塌（Semantic Collapse）」，影片中極短暫但關鍵的造謠片段會被周圍冗長、無害的背景內容在數學上稀釋，進而使得相似度比對失效 。

第二種則是離散嵌入聚合（Discrete Embedding Aggregation），結合 Elasticsearch 9 的巢狀文件（Nested Documents）結構。在此架構下，影片的詮釋資料（Metadata）作為父文件（Parent Document），而每一個 75 秒的片段向量則作為子文件（Child Documents）獨立儲存 。當有新的 30 秒短影音傳入 Cofacts 系統時，查詢向量會與資料庫中所有獨立的片段向量進行相似度計算。只要該短片與某支 5 分鐘長片中的「第二個片段」產生高度的幾何契合，Elasticsearch 就能精準地將該長片檢索出來。這種策略完整保留了影音序列的微觀語意顆粒度，是處理 5 分鐘以上長篇謠言影片的唯一最佳解法 。

## **Elasticsearch 9 混合搜尋與近似重複排名架構**

當十萬筆資料全數轉換為 Gemini 向量並存入 Elasticsearch 後，當新訊息進入時，必須實作同時融合 TF-IDF 與語意向量的混合搜尋（Hybrid Search）。這不僅是為了提升檢索召回率（Recall），更是為了彌補純向量搜尋在事實查核領域的固有缺陷。

### **結合詞彙精確度與語意廣度**

單純依賴向量相似度（Semantic Search）雖然能夠解決同義詞替換與跨模態映射的問題，但向量壓縮技術在面對極為罕見、精確的專有名詞（例如特定的化學物質名稱、法案字號、人名等）時，容易發生精確度喪失的現象 。相對地，傳統的詞彙搜尋（Lexical Search，如基於 TF-IDF 演進而來的 BM25 演算法）雖然無法理解意圖，但對於精確關鍵字的匹配卻擁有絕對的優勢 。

Elasticsearch 9 透過混合搜尋將兩者的優勢結合。當一則新謠言進入時，系統會同時觸發 BM25 的詞彙比對與 HNSW（階層可導航小世界）演算法的近似最近鄰（ANN）向量比對 。這確保了搜尋結果既能在語意上捕捉到被抽換詞面的謠言變體，又能在包含特定專有名詞時精準鎖定目標。

### **排名演算法的最佳化：線性檢索器對決 RRF**

Cofacts 對於搜尋結果的呈現有一個極為明確的商業邏輯要求：**檢索結果必須嚴格依照相似度排序，近似重複（Near-duplicate）的內容必須絕對置於列表最頂端，其次才是主題相同的內容**。為了實現這個目標，在 Elasticsearch 中合併兩種搜尋結果的演算法選擇至關重要。

目前 Elasticsearch 提供兩種主流的融合機制：倒數秩融合（Reciprocal Rank Fusion, RRF）與線性檢索器（Linear Retriever）。

RRF 是一種無監督的排序演算法，其運作原理是完全捨棄 BM25 與向量比對的原始絕對分數（Absolute Scores），僅根據文件在兩個檢索列表中的排名（Rank）進行重新計分 。雖然 RRF 易於實作且不需要校準參數，但它極度不適合 Cofacts 的需求。因為 RRF 忽略了分數幅度，如果兩份文件在各自的列表中排名第一，RRF 會賦予它們相同的權重；這導致一個餘弦相似度高達 0.99 的絕對近似重複影片，可能會與一個相似度僅有 0.70 但碰巧在另一個查詢中排名第一的粗略相關文件獲得相近的最終排名，徹底破壞了將 Near-duplicate 置於頂端的防線 。

因此，系統必須強制採用 **線性檢索器（Linear Retriever）配合 MinMax 正規化（Normalization）** 。線性檢索器會保留各個檢索器的原始分數幅度，並透過加權總和（Weighted Sum）來決定最終排名 。由於 BM25 的分數是無上限的（Unbounded，範圍從零到理論無限大），而 Gemini 向量的餘弦相似度介於 0 與 1 之間，因此必須透過 MinMax 正規化將兩者的分數強制作弊到相同的 區間 。

在實作時，開發團隊可以賦予向量檢索器極高的權重（例如將向量相似度的權重設為 0.8，TF-IDF/BM25 的權重設為 0.2）。在這種極端不對稱的加權機制下，只要新傳入的圖片或影片與資料庫中的某個項目在 Gemini 的多模態空間中產生幾何上的高度重合（分數接近 1.0），其總和分數將無條件碾壓其他僅有部分關鍵字重合的文件。這完美確保了近似重複內容永遠佔據搜尋結果的頂端，而跨模態的特性則保證了不論是文字、影片還是截圖，只要語意內涵一致，皆能在同一個列表中按相似度降冪排列 。

### **硬體資源與記憶體消耗評估**

十萬筆資料的 3,072 維度向量儲存，在 Elasticsearch 的系統架構下是非常輕量級的負載。每個 3,072 維度的 float32 向量大約佔用 12.2 KB 的記憶體。十萬筆文件僅需約 1.22 GB 的原始儲存空間。為了支撐極速的向量搜尋，Elasticsearch 需將 HNSW 圖結構載入記憶體中，這大約會增加每個向量 3% 到 5% 的額外開銷（用於儲存節點連接的指標）。綜合計算下來，整個 Cofacts 向量索引（Index）在 JVM 堆外記憶體（Off-heap Memory）的佔用量將遠低於 2 GB 。這意味著現有的伺服器基礎設施無需進行昂貴的硬體擴充或導入 GPU 加速，即可毫無延遲地支撐這種規模的混合檢索作業。

## **經濟可行性與十萬筆資料遷移成本分析**

要將累積十年的十萬筆多模態資料全部透過 Gemini Embedding 2 轉換為向量存回 Elasticsearch，必須仔細評估 Google Cloud Vertex AI 的計費結構。該 API 針對不同的資料輸入模態有著極為精確的微切分計費標準，且生成嵌入向量時，不會產生任何輸出標記（Output Tokens）的費用 。

### **Vertex AI 多模態嵌入基礎費率**

根據 gemini-embedding-2-preview 模型的公開價格表，標準的即時線上推論（Online Inference）費率如下 ：

* **純文字與網頁內容：** 每一百萬（1M）個輸入標記收費 $0.20 美元。  
* **圖片與截圖：** 每張圖片收費 $0.00012 美元。  
* **音訊（純音檔）：** 每秒收費 $0.00016 美元。  
* **影片（無聲/純影像）：** 每個採樣影格收費 $0.00079 美元（預設採樣率為每秒 1 影格，即每秒 $0.00079 美元）。  
* **帶有音軌的影片：** 系統會同時收取視訊影格與音訊的處理費用，總計為每秒 $0.00079 \+ $0.00016 \= $0.00095 美元 。

值得注意的是，針對歷史資料的大規模一次性轉換，Google 提供了批次預測 API（Batch Prediction API）。對於這種非急迫性、可以接受 24 小時內異步處理完畢的大規模任務，批次 API 提供高達 **50% 的費率折扣** 。Cofacts 必須利用此批次機制來執行十萬筆資料的初始向量化。

## **效能基準測試與多模態錯誤資訊防禦能力**

從學術與工程基準測試的角度來看，引進原生多模態嵌入模型是防禦複雜錯誤資訊（Misinformation）的必然趨勢。在處理真實世界中經常伴隨跨語系（例如中英文字幕混合）與局部影像竄改的假新聞時，傳統的單模態系統經常出現推理斷層 。

Gemini Embedding 2 在各大基準測試中展示了卓越的防禦潛力。在涵蓋超過 250 種語言、一百多項任務的巨量多語系文本嵌入基準測試（MMTEB）中，它大幅超越了過往的最新技術標準，並且在低資源語言的跨語系檢索（Cross-lingual Retrieval）上也表現優異 。這對於經常需要跨國查核、或是處理多語系夾雜的 Cofacts 平台而言是一大助力。更重要的是，由於模型能理解同一語意在「視覺文字（截圖）」、「音訊頻譜（朗讀）」與「數位編碼（純文字）」中的關聯，當惡意行為者企圖透過改變傳播載體來規避查核引擎時，該系統能夠自動在統一的潛在空間中將其攔截 。這種能力將使 Cofacts 從過往「被動應對變種」的困境，轉化為「主動語意收斂」的前瞻防禦姿態。

## **結論與部署藍圖**

綜上所述，將 Gemini Embedding 2 搭配 Elasticsearch 9 用於 Cofacts 資料庫的升級，不僅在技術上完全可行，在經濟效益上更具備壓倒性的優勢。這個系統架構典範轉移將徹底解決 Cofacts 長期面臨的跨模態辨識斷層、AI 變造影片難以追蹤，以及詞面抽換導致 TF-IDF 失效的核心痛點 。

為了確保系統上線後的精準度，工程團隊在實作上應嚴格遵循以下藍圖部署：

1. **實作巢狀文件切割策略：** 面對 30 秒至 5 分鐘不等的不規則長度影片，堅決放棄平均池化，採用 75 秒滑動窗口進行切割，並利用 Elasticsearch 的 nested 結構儲存離散向量，以確保最微小的變造片段也能被捕捉 。  
2. **停用 RRF，擁抱正規化線性檢索器：** 為了滿足「近似重複內容必須排在最前方」的關鍵業務邏輯，必須設定帶有 MinMax 正規化的 Linear Retriever，並賦予語意向量壓倒性的權重配比，以此建立嚴密的幾何相似度過濾網 。  
3. **善用批次處理降低成本：** 透過 Vertex AI Batch API 執行十萬筆歷史資料的遷移，將一次性建置成本控制在 500 美元以內 。  
4. **維持全維度向量：** 在 10 萬筆資料量級下，系統記憶體負載極低，應維持 3,072 輸出維度以保留模型最強大的多模態區分能力 。

這套架構將賦予 Cofacts 平台前所未有的「全知視角」，不論錯誤資訊披上何種模態的外衣，系統皆能在毫秒之間，從深邃的數學空間中揭露其真實的語意本質。
