#### API
> 想確認如何實作 [name=mrorz]
> 時機：CreateMediaArticle 內 (建立 article doc)，產出 transcript 後，希望寫入 ydoc.

rumors-api 直接產生 ydoc v.s. 在 collab server 實作一個 API？

- 目前寫入 article.text 與 ydoc 的地方：https://github.com/cofacts/collab-server/blob/master/hocuspocus-extension-elasticsearch/src/elasticsearch.ts
    - 使 `hocuspocus-extension-elasticsearch` depend on Cofacts-specifc `articles` index
    - 似乎應該移動到 collab-server 的 [server hooks](https://tiptap.dev/hocuspocus/server/hooks) [name=mrorz]
      - [onStoreDocument](https://tiptap.dev/hocuspocus/server/hooks#on-store-document)
      - 想做在前端 create snapshot。如果使用者不想存檔，可以按取消，此時不應該 update [name=nonumpa]
      - ydoc 還是會寫入，但 article.text 會在按完成時更新 [name=nonumpa]
      - 討論結果：因為會 cancel 到其他人的 work，不符合大家使用 google doc 等共編的體驗，還是改回原本的自動 snapshot (onStoreDocument 時)
- Option 1：直接寫入 ydoc
    - 直接用 Y.js 寫一個 ydoc 文件
    - no structured data？
    - 現在 ydoc 格式是 prosemirror 的 XML 格式 [name=nonumpa]
- Option 2：呼叫 collab-server API
    - 可能比較乾淨，只有一個地方會同時寫入 ydoc 並將文字放進 article text
    - rumors-api 與 collab-server 需要有個 shared secret
- Option 3：rumors-api 跟 collab-server 放進 mono-repo 然後抽出寫 ydoc 的邏輯共用
    - 前置作業一大堆⋯⋯
