#### Auth
> [name=mrorz]
> 

難點：
- api.cofacts.tw 與 collab.cofacts.tw 網域不同，cookie 不互通。
- 因此，瀏覽器連線到 collab.cofacts.tw 時，不會送出 api.cofacts.tw 網域的 login cookie。

目前想到的大方向：
1. rumors-site 連線到 rumors-api
2. api 使用 cookie 拿到 user ID 之後傳給 collab-server

具體作法選擇：
1. api.cofacts.tw 的 login cookie 改成放到 .cofacts.tw domain
    - production / staging 應該可以運作，但 dev 環境無法⋯⋯
    - collab.cofacts.tw 也要有解析 API session cookie 的能力
2. collab-server 與 rumors-api 合併
    - rumors-api 開一個 endpoint 接 websocket（如 api.cofacts.tw/collab ）
    - 透過既有 authentication 機制驗證 login session cookie、寫入 user ID 給 hocuspocus
    - 使用 hocuspocus server 的 `handleConnection()`
    - https://tiptap.dev/hocuspocus/server/examples#koa
    - 可以少管理一台機器 XD
3. rumors-api proxy websocket to hocuspocus server
    - rumors-api 開一個 endpoint （如 api.cofacts.tw/collab ）做成 proxy
        - 參考 https://github.com/http-party/node-http-proxy
        - koa: 參考[koa-proxy 如何使用 node-http-proxy](https://github.com/raphaelsoul/koa-proxies/blob/master/index.js#L84)再[另外聽 upgrade event](https://github.com/vagusX/koa-proxies/issues/1#issuecomment-546778781)
    - 透過既有 authentication 機制驗證 login session cookie

大方向 alternative:
- rumors-api 開一個 endpoint 把 login session 做成超短效期 JWT token
- 進入編輯頁面時，呼叫該 endpoint 產出 JWT token，在往 hocuspocus server 的連線 request 中帶入該 token
- hocuspocus server 在 onAutenticate 時，解析 token 拿 user ID。（斷線後就無效。）
